== Excerpt

When you hit "run," a remarkable chain of transformations quietly takes place before your code ever touches the processor. A compiler's job is to bridge the enormous gap between human-readable source code and machine instructions, doing so through a series of well-defined stages: lexical analysis, parsing, semantic analysis, optimization, and code generation. Each phase peels away another layer of abstraction.

What's fascinating is how much work happens before a single instruction is executed. The compiler builds symbol tables, constructs abstract syntax trees, resolves types, and often reorganizes your logic entirely in pursuit of efficiency. The code that actually runs on your CPU can look remarkably different from what you wrote — and that's entirely by design.

== Phase One: Lexical Analysis

The first phase, performed by the _lexer_ (or scanner), reads the raw character stream and groups characters into meaningful units called _tokens_. Keywords, identifiers, literals, operators, and punctuation each become a distinct token type. Whitespace and comments are typically discarded at this stage. The output is a flat token stream that subsequent phases can process without worrying about individual characters.

== Phase Two: Parsing and the Syntax Tree

The _parser_ consumes the token stream and checks that it conforms to the language's grammar, typically expressed as a context-free grammar. If the grammar is satisfied, the parser produces an _Abstract Syntax Tree_ (AST) — a hierarchical structure that captures the grammatical relationships between tokens. An `if` statement, for instance, becomes a node with three children: the condition expression, the then-branch, and the optional else-branch. Syntax errors surface here when the token stream violates grammatical rules.

== Phase Three: Semantic Analysis

With a syntactically valid AST in hand, the compiler performs _semantic analysis_. This phase enforces rules that grammars cannot express: type checking, scope resolution, and checking that variables are declared before use. Type inference engines, where present, also operate here. The result is an annotated AST — the same tree structure, but enriched with type information and symbol table references.

== Phase Four: Intermediate Representation and Optimisation

Most modern compilers translate the annotated AST into an _Intermediate Representation_ (IR) — a lower-level, language-agnostic form that is easier to analyse and transform. LLVM's IR is a well-known example. The IR is then subjected to a series of _optimisation passes_: dead code elimination, constant folding, inlining, loop unrolling, and many others. Optimising the IR rather than the source or machine code keeps the optimisation logic independent of both the source language and the target architecture.

== Phase Five: Code Generation

The final phase translates the optimised IR into target machine code or bytecode. A register allocator maps IR variables to physical CPU registers, instruction selection chooses the best machine instructions, and an instruction scheduler reorders operations to exploit CPU pipelining. The output may be native machine code, JVM bytecode, WebAssembly, or any other target the compiler supports.

== Why This Matters for Developers

A working knowledge of compiler phases explains many seemingly mysterious behaviours: why an unused variable causes a warning (semantic analysis), why certain loop structures are faster than others (IR optimisation), and why template-heavy C++ compiles slowly (AST explosion). It also provides the conceptual foundation for building domain-specific languages, linters, and static analysis tools.

== Conclusion

Compilers are among the most sophisticated software artefacts ever built, yet their internal architecture follows a clear, logical sequence of transformations. Each phase has a well-defined input and output, and each one adds a layer of understanding — from raw text to executable logic — that ultimately makes high-level programming possible.
