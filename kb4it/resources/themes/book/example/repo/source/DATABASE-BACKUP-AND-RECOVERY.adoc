== Overview

This reference covers the key concepts, strategies, and trade-offs in database backup and recovery planning. It applies to relational databases in general; specific tools and commands vary by engine.

== Backup Types

=== Full Backup

A complete copy of the entire database at a point in time.

* *Pros:* simplest to restore; no dependency on other backups.
* *Cons:* highest storage cost; longest backup window.

=== Incremental Backup

Captures only changes since the last backup (full or incremental).

* *Pros:* minimal storage; fast to produce.
* *Cons:* restore requires applying the full backup followed by each incremental in sequence; longer recovery time.

=== Differential Backup

Captures all changes since the last full backup.

* *Pros:* faster restore than incremental (only two steps: full + latest differential).
* *Cons:* grows larger over time until the next full backup is taken.

=== Transaction Log Backup

Backs up the transaction log, enabling point-in-time recovery.

* *Pros:* fine-grained recovery; minimises data loss.
* *Cons:* requires a database in full recovery model; log files must be retained in sequence.

== Recovery Point Objective and Recovery Time Objective

Two metrics define backup strategy requirements:

* *RPO (Recovery Point Objective)* — the maximum acceptable data loss, expressed as time. An RPO of one hour means losing up to one hour of transactions is acceptable.
* *RTO (Recovery Time Objective)* — the maximum acceptable time to restore service after a failure.

A backup strategy must satisfy both constraints. Frequent transaction log backups reduce RPO; pre-staged standby environments reduce RTO.

== Backup Verification

A backup that has never been tested is not a backup — it is a hope. Verify backups by:

. Restoring to an isolated test environment on a regular schedule.
. Confirming data integrity (row counts, checksums, application smoke tests).
. Measuring actual restore time against the RTO target.

Document every test and its outcome.

== Retention Policy

[cols="1,3,2", options="header"]
|===
| Backup Type | Retention | Frequency
| Full | 4 weeks | Weekly
| Differential | 2 weeks | Daily
| Transaction log | 7 days | Every 15–60 minutes
|===

Adjust retention to meet RPO requirements and storage budget constraints.

== Notes

CAUTION: Never store all backup copies on the same storage system as the database. A storage failure that corrupts the database will corrupt the backups at the same time.

NOTE: Encrypt backup files at rest and in transit. An unencrypted database backup is as sensitive as the database itself.
